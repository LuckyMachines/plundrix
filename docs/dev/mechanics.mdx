# Actions & Mechanics

Each round, every player chooses exactly one action. Actions resolve simultaneously ‚Äî no player has information about others' choices when submitting.

---

## Pick (Action = 1)

Attempt to crack the next lock on the vault.

**Success formula:**

```
chance = min(40 + tools * 15, 95)
```

| Tools | Success Rate |
|-------|-------------|
| 0 | 40% |
| 1 | 55% |
| 2 | 70% |
| 3 | 85% |
| 4+ | 95% |

**If stunned:** Automatically fails. No roll is made. Returns `PICK_FAILED_STUNNED`.

**On success:** `locksCracked++` and emits `LockCracked`. If `locksCracked >= 5`, the game ends.

**Strategy notes:**
- At 0 tools, PICK is a coin flip. Search first unless you're behind.
- At 3+ tools, PICK is strongly favored. Stop searching and start cracking.
- The 95% cap means there's always a 5% chance of failure, even fully tooled.

---

## Search (Action = 2)

Search a compartment for lockpicking equipment.

**Success rate:**
- Normal: **60%**
- Stunned: **30%**

**On success:** `tools++` (if below `MAX_TOOLS = 5`) and emits `ToolFound`.

**Failure modes:**
- `SEARCH_FAILED_ROLL` ‚Äî RNG miss
- `SEARCH_FAILED_MAX_TOOLS` ‚Äî Already at 5 tools

**Strategy notes:**
- Search is the investment action ‚Äî tools compound your PICK success rate.
- Even when stunned, Search still has a 30% chance (unlike PICK which auto-fails).
- Diminishing returns: going from 3‚Üí4 tools only gains +10% pick chance, but costs a round.

---

## Sabotage (Action = 3)

Target another player. Always succeeds (no RNG). Costs your entire action for the round.

**Effects:**
1. **Stun** ‚Äî Target is stunned for the *next* round (not the current one)
2. **Tool theft** ‚Äî If target has tools and attacker is below max, steal one tool

**Outcomes:**
- `SABOTAGE_SUCCESS_STEAL` ‚Äî Stunned target + stole a tool
- `SABOTAGE_SUCCESS_STUN_ONLY` ‚Äî Stunned target, but attacker already at max tools
- `SABOTAGE_SUCCESS_NO_TOOL` ‚Äî Stunned target, but target had no tools to steal

**Strategy notes:**
- Sabotage is the defensive/aggressive action. Use it to slow down the leader.
- The tool theft is a two-swing: you gain one, they lose one. A 2-tool net shift.
- Costs you a round of progress ‚Äî only worth it if the target is threatening to win.
- Multiple players can sabotage the same target in one round (stun doesn't stack, but each attacker can steal a tool).

---

## Stun Mechanic

Stun is the core disruption mechanic. It lasts exactly one round.

**Timeline:**
```
Round N: Player A sabotages Player B
         ‚Üí B's PICK/SEARCH resolves normally this round
         ‚Üí B gets stunned flag set

Round N+1: B is stunned
           ‚Üí PICK auto-fails
           ‚Üí SEARCH success drops to 30%
           ‚Üí Stun clears at end of resolution

Round N+2: B is no longer stunned
```

**Key details:**
- Stun is applied in **Phase 3** of resolution (after PICK/SEARCH in Phase 1)
- Existing stuns are cleared in **Phase 2** (between PICK/SEARCH and SABOTAGE)
- This means stun always affects the *next* round, never the current one
- A player can be sabotaged by multiple opponents ‚Äî the stun doesn't stack, but each can steal a tool

## Default Moves (AFK Handling)

When `resolveRound()` is called after timeout and a player hasn't submitted:

1. Player is assigned `Action.PICK` with `sabotageTarget = address(0)`
2. `DefaultActionAssigned` event is emitted
3. The PICK resolves normally through the standard resolution pipeline

**Why PICK and not another action?**
- PICK is the only action that can end the game (crack locks)
- SEARCH only accumulates tools ‚Äî slower convergence
- SABOTAGE requires choosing a target ‚Äî any deterministic selection creates unfair bias

This ensures every game will eventually complete, even with fully AFK players.

## Randomness

Each action's success is determined by pseudorandom number generation:

```solidity
uint256 rand = uint256(keccak256(abi.encodePacked(
    blockhash(block.number - 1),
    gameID,
    round,
    block.timestamp,
    entropy,   // external entropy if provided
    seed       // player index
))) % 100;
```

The result is a number 0-99, compared against the action's success threshold.

**External entropy:** When `requireExternalEntropy` is enabled, the `RANDOMIZER_ROLE` must call `provideRoundEntropy()` before a round can resolve. This entropy is sourced from the [drand](https://drand.love/) randomness beacon.

**Security note:** On-chain randomness is not perfectly secure against block producers. For high-value STAKES games, external entropy from drand provides stronger guarantees. See the [mainnet runbook](/docs/mainnet-runbook) for risk analysis.
